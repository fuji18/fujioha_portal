# 公開後チェックリスト

本番公開の完了直後から 1 週間程度までに確認する項目をまとめる。対象は `apps/portal` / `apps/hokkaido` / `apps/game` の各 Cloudflare Pages 配信と、それに紐づく検索・計測・運用である。

## 使い方

- まず公開当日に「公開直後」を上から順に実施する
- 重大な異常があれば新規改善より先に切り戻しまたは応急対応を行う
- 初日で問題がなければ「24時間以内」「1週間以内」を順に消化する
- 実施メモは PR、Issue、運用ノートのいずれかに残し、属人化を防ぐ

---

## 公開直後

### 配信と表示

- [ ] 本番 URL で表示できる (`https://fujioha.com` 配下の各公開先、または対応サブドメイン)
- [ ] 主要ページで 200 が返る
- [ ] 404 ページが想定どおり表示される
- [ ] 直前に更新したコンテンツが反映されている
- [ ] モバイル幅とデスクトップ幅の両方でレイアウト崩れがない

### Cloudflare Pages / キャッシュ

- [ ] 対象プロジェクトの本番デプロイが成功している
- [ ] 期待したブランチが production として配信されている
- [ ] 古いアセットが配信されていない
- [ ] 再読み込み後も JS/CSS 取得エラーが出ない
- [ ] `_headers` で意図したキャッシュ・セキュリティヘッダが載っている

### 導線とリンク

- [ ] ヘッダー、フッター、カード一覧の主要リンクが切れていない
- [ ] 外部リンクに `target="_blank"` を使う箇所は `rel="noopener noreferrer"` が付いている
- [ ] `apps/game` と `apps/hokkaido` の featured/recent 一覧に期待した項目が出ている
- [ ] OGP 画像、タイトル、説明文が主要ページで意図どおりになっている

### 配信ファイル

- [ ] `sitemap.xml` が取得できる
- [ ] `rss.xml` / `atom.xml` を提供しているアプリは正常に取得できる
- [ ] `robots.txt` の内容が本番向けになっている
- [ ] `apps/game` の `service-worker.js` / `sw.js` 更新後に配信不整合が起きていない

---

## 24時間以内

### 監視とログ

- [ ] 404 / 500 / 403 の増加がないかログまたは分析基盤で確認した
- [ ] クライアントエラーを収集している場合、重大エラーが出ていない
- [ ] 問い合わせ、SNS、フォーム経由の初期不具合報告を確認した
- [ ] ロールバック手順が現時点でも実行可能か確認した

### 計測

- [ ] アクセス計測が本番で動いている
- [ ] 主要流入元と主要ページの閲覧数を確認した
- [ ] 想定していた導線到達率を計測できている
- [ ] 計測タグ重複や二重送信がない

### SEO

- [ ] Google Search Console / Bing Webmaster Tools に登録済みならクロール異常がない
- [ ] `sitemap.xml` を送信または再送した
- [ ] `canonical` が重複や誤設定になっていない
- [ ] noindex にしたいページだけが noindex になっている

---

## 1週間以内

### パフォーマンス

- [ ] 主要ページの Lighthouse を再計測した
- [ ] Core Web Vitals の悪化がない
- [ ] 画像・フォント・スクリプトの転送量が想定内である
- [ ] 低速回線や実機スマホでも致命的な体感劣化がない

### コンテンツ品質

- [ ] 誤字、文言揺れ、リンク先の不一致を修正した
- [ ] featured/recent/feed の並びが公開意図と一致している
- [ ] 日付、公開状態、URL などコンテンツ定義のメタデータに齟齬がない

### セキュリティ

- [ ] 不要な公開ファイルが含まれていない
- [ ] CSP やその他ヘッダが運用実態に合っている
- [ ] secrets やトークンがリポジトリ・ビルドログ・フロントエンドへ露出していない
- [ ] 依存パッケージの既知脆弱性を確認した

---

## 継続運用

- [ ] 毎週または隔週で 404 と主要離脱ページを見直す
- [ ] 月次で検索流入、人気ページ、CV 相当導線を確認する
- [ ] 反応が薄い導線は文言・配置・クリエイティブを改善する
- [ ] 公開後に見つかった再発バグは playbook またはテストへ反映する
- [ ] 新しい配信手順やハマりどころは `docs/` に追記する

---

## 障害時の最小対応

1. 症状の再現 URL と発生時刻を記録する
2. 影響範囲を切り分ける（1 アプリのみか、全体か、キャッシュか）
3. 直近の変更、Cloudflare Pages のデプロイ履歴、関連 PR を確認する
4. 切り戻し可能なら先に復旧し、原因調査はその後に回す
5. 再発防止としてドキュメント、監視、テストのどれで防ぐかを決める

## このリポジトリ向けの最低ライン

次の 6 項目は毎回必須とする。

- [ ] Cloudflare Pages の production deployment 成功確認
- [ ] 主要 URL の表示確認
- [ ] `sitemap.xml` / `robots.txt` / feed 系 URL の確認
- [ ] featured/recent/card 一覧の目視確認
- [ ] 計測とエラーログの初期確認
- [ ] ロールバック可否の確認
